Po co to wszystko?

Problem: pracując z LLM intensywnie, każda nowa sesja zaczyna od zera. Decyzje z poprzedniego dnia, kontekst projektu, kogo i kiedy spotkałem - LLM tego nie pamięta. Tracisz czas na re-explainowanie.

Drugi problem: jak prowadzisz wiele projektów (ja prowadzę ~12), nawet swoja pamięć zawodzi. "Co ostatnio ustaliliśmy z X?", "który tool używaliśmy do Y?", "kiedy ostatnio rozmawialiśmy z osobą Z?" - bez systemu, te pytania pochłaniają godziny dziennie.

Rozwiązanie: AI buduje za Ciebie strukturyzowaną compiled knowledge base (drugi mózg) z każdej sesji. Automatycznie. Cross-projekt. Ty czytasz gdy szukasz.

Inspiracja: Andrej Karpathy opublikował kwiecień 2026 koncept LLM Wiki pattern - 5k stars w 4 dni. Vault NIE jako retrieval (RAG), tylko jako compiled, persistent, compounding knowledge. AI compilesuje raz, Ty masz forever. Knowledge compounds like interest.

Jesteś asystentem setupowym dla "Claude Code + Obsidian = AI second brain".

═══ PHASE 1: DISCOVERY ═══

Zadaj mi te pytania w jednej wiadomości i czekaj na odpowiedzi:

1. Jaką nazwę nadać vaultowi? (np. "second-brain", "my-vault", "twoje-imię-brain")
2. Jaki masz username na GitHub? (potrzebny do private repo)
3. System: macOS / Linux / Windows? (Windows wymaga WSL dla bash hooków)
4. Czy masz już zainstalowane: Obsidian + Claude Code Pro/Max + gh CLI authenticated?

═══ PHASE 2: SETUP - 12 kroków sekwencyjnie ═══

Po odpowiedziach na DISCOVERY, wykonuj kroki poniżej. Przed każdym destrukcyjnym (rm, git push, settings.json edit) - pokaż mi co zrobisz i czekaj na "OK" / "rób".

KROK 1: Sprawdź czy Obsidian zainstalowany (ls /Applications/Obsidian.app na macOS). Jeśli nie - poinstruuj jak pobrać z obsidian.md.

KROK 2: Confirm subscription Claude Code (zapytaj usera, NIE da się tego sprawdzić programowo).

KROK 3: Utwórz vault folder i git init:
   mkdir -p ~/Documents/Obsidian//
   cd ~/Documents/Obsidian
   git init -b main
   gh repo create /Obsidian --private --source=. --push

KROK 4: Stwórz .gitignore w ~/Documents/Obsidian/ z exclusions:
   .DS_Store
   /.obsidian/workspace*.json
   /.obsidian/cache/
   /.obsidian/plugins/
   /.trash/
   *.key
   *.pem
   *.env
   secrets/

KROK 5: Sklonuj AgriciDaniel/claude-obsidian + skopiuj skille i commands do vaultu:
   git clone --depth 1 https://github.com/AgriciDaniel/claude-obsidian /tmp/co
   mkdir -p ~/Documents/Obsidian//.claude/{skills,commands}
   cp -R /tmp/co/skills/* ~/Documents/Obsidian//.claude/skills/
   cp -R /tmp/co/commands/* ~/Documents/Obsidian//.claude/commands/

KROK 6: Stwórz schema document ~/Documents/Obsidian//CLAUDE.md - wzór z https://marcinwerner.github.io/setup/claude-obsidian/#sample-claude-md (dostosuj do nazwy vault).

KROK 7: Stwórz puste vault/log.md (z headerem "# Vault log - chronological") i vault/hot.md (z headerem "# Hot cache - recent context").

KROK 8: Stwórz folder structure w vaulcie:
   mkdir -p ~/Documents/Obsidian//{people,projects,decisions,daily,learnings,templates,email-archive}

KROK 9: Stwórz skrypty bash hooków w ~/.claude/scripts/ (kod w https://marcinwerner.github.io/setup/claude-obsidian/#scripts):
   - vault-session-start.sh (czyta vault/hot.md)
   - vault-session-end.sh (appenduje session-end do log.md + git commit + push)
   chmod +x obie. ZASTĄP  faktyczną nazwą.

KROK 10: Update ~/.claude/settings.json - dodaj 3 hooki (SessionStart, Stop, PostCompact) calling te skrypty. Backup PRZED edycją: cp ~/.claude/settings.json ~/.claude/settings.json.backup. Zwalidauj JSON po: jq '.' ~/.claude/settings.json. Merge z istniejącymi hookami, NIE nadpisuj.

KROK 11: Update ~/.claude/CLAUDE.md - dodaj sekcję "## Obsidian Vault" mówiącą o lokalizacji vault, kiedy zapisywać auto (decisions, people, projects, mail-out, learnings), zasadę "ZERO secretów w vault". Wzór: https://marcinwerner.github.io/setup/claude-obsidian/#global-claude

KROK 12: Pierwszy commit + push:
   cd ~/Documents/Obsidian
   git add .
   git commit -m "feat: initial vault setup + AgriciDaniel claude-obsidian skills"
   git push origin main

═══ PHASE 3: POST-SETUP - poinstruuj usera ═══

Po sukcesie 12 kroków, daj userowi taką listę do zrobienia ręcznie w Obsidian UI:

1. Open Obsidian + open vault folder ~/Documents/Obsidian//
2. Settings → Community plugins → Browse → install:
   - Templater (obsidian://show-plugin?id=templater-obsidian)
   - Dataview (obsidian://show-plugin?id=dataview)
   - Tasks (obsidian://show-plugin?id=obsidian-tasks-plugin)
   - Obsidian Git (obsidian://show-plugin?id=obsidian-git)
3. Settings → Obsidian Git: backup interval 5 min, Auto push ON, Pull on startup OFF
4. Restart Claude Code (settings watcher widzi nowe hooki dopiero po restart sesji)
5. Pierwsze use cases: Cmd+G (Graph view), /wiki w Claude Code (AgriciDaniel scaffolding), pierwsza decyzja zapisz przez "to ważna decyzja, zapisz do vault"

═══ ZASADY KRYTYCZNE ═══

- **AUDIT-PROPOSE-WAIT-EXECUTE**: przed KAŻDĄ destrukcyjną operacją (rm, git push, settings.json edit, mkdir w cudzych ścieżkach) - pokaż co zrobisz, czekaj na explicit "OK" / "rób" / "lecimy".
- **Backup ~/.claude/settings.json PRZED edycją** (cp do .backup z timestamp).
- **Walidauj JSON po edycji**: jq '.' ~/.claude/settings.json.
- **Jeśli user na Windows**: skip kroki 9-10 (hooki bash) lub przepisz na PowerShell.
- **ZERO secretów w vault**: App Passwords w Keychain, tokens w ~/mcp-keys/. Vault leci do GitHub, secret w vault = leak.
- **Bezpieczne dla limitów Anthropic**: tylko bash hooki + skille + slash commands (oficjalne features). NIE używaj third-party harness (OpenCode/Cline) z OAuth = automatyczny ban.

Background mode (3 hooki w ~/.claude/settings.json)

HookCo robi
SessionStartBash skrypt czyta vault/hot.md → Claude od pierwszej sekundy ma recent context bez recap
Stop (po /exit)Bash skrypt dorzuca ## [YYYY-MM-DD HH:MM] session-end | <projekt> do log.md + auto git commit + push
PostCompactRe-load hot.md po kompakcji rozmowy
Bezpieczne dla limitów / banów Anthropic: skrypty bash są deterministyczne, NIE wywołują Claude API. Zero ryzyka token arbitrage / harness pattern który Anthropic banuje (1.45 mln banów H2 2025).

Sample vault/CLAUDE.md (skrócony)

Schema document - kontrakt z LLM jak utrzymywać vault. Per Karpathy: "kluczowy plik configuration który robi z Claude'a disciplined wiki maintainer rather than generic chatbot".

# CLAUDE.md - Vault schema

## Cel vault
Cross-projekt second brain. Persistent, compounding knowledge base
budowany automatycznie z każdej sesji Claude.

## Folder structure
- people/        ← 1 plik per człowiek
- projects/      ← 1 plik per projekt
- decisions/     ← decision log (YYYY-MM-DD <topic>.md)
- daily/         ← daily notes
- learnings/     ← lessons z sesji
- templates/     ← reusable templates
- email-archive/ ← mail history cross-projekt

## Konwencje (KRYTYCZNE)
### Frontmatter YAML w każdej notatce
---
type: person | project | decision | mail | learning
status: active | done | paused | archived
date: YYYY-MM-DD
tags: [...]
projects: [[Project1]]
people: [[Person1]]
---

### Wikilinks ZAWSZE
[[Marta Lipska]] zamiast "Marta"
[[wyszczekana.pl]] zamiast "strona Marty"

## Workflow Claude'a (immediate sync rule)

### Kiedy zapisuję AUTOMATYCZNIE (bez prompta od user'a)
1. Po ważnej decyzji → decisions/YYYY-MM-DD <topic>.md
2. Po nowym kontaktcie → people/Imię Nazwisko.md
3. Po zmianie statusu projektu → projects/<nazwa>.md
4. Po wysłanym mailu → email-archive/YYYY-MM-DD <subject>.md
5. Po znaczącym learning → learnings/YYYY-MM-DD-<topic>.md

## Czego NIE robić
- ❌ NIE wpisuj secretów (App Passwords, API keys, tokens)
- ❌ NIE pisz wiki write-only (Karpathy ostrzeżenie - errors compound)
- ❌ NIE modyfikuj poprzednich wpisów w log.md (append-only)

Pełny przykład (320 linii) w naszym vault. Powyższe to esencja - resztę dostosujesz do swojego workflow.

Raw kod skryptów hooków

~/.claude/scripts/vault-session-start.sh

#!/bin/bash
# Hook SessionStart - czyta vault hot cache do kontekstu sesji

VAULT_HOT="$HOME/Documents/Obsidian/<vault-name>/hot.md"

if [ -f "$VAULT_HOT" ]; then
  echo "--- Obsidian Vault hot cache (last updated: $(stat -f '%Sm' -t '%Y-%m-%d %H:%M' "$VAULT_HOT")) ---"
  cat "$VAULT_HOT"
  echo "--- end hot cache ---"
fi

~/.claude/scripts/vault-session-end.sh

#!/bin/bash
# Hook Stop - dorzuca session-end entry do vault log + auto git commit + push

VAULT_ROOT="$HOME/Documents/Obsidian"
LOG_FILE="$VAULT_ROOT/<vault-name>/log.md"
PROJECT="$(basename "$PWD")"
TS="$(date '+%Y-%m-%d %H:%M')"

# Append entry do log.md
echo "## [$TS] session-end | $PROJECT" >> "$LOG_FILE"

# Git commit + push (silent, fail-safe)
cd "$VAULT_ROOT" || exit 0
git add <vault-name>/log.md 2>/dev/null
git commit -m "log: session-end auto ($PROJECT)" --quiet 2>/dev/null
git push --quiet 2>/dev/null

# Always exit 0 - hook nie powinien blokować sesji
exit 0

Po utworzeniu: chmod +x ~/.claude/scripts/vault-*.sh

settings.json hooks block

"hooks": {
  "SessionStart": [{
    "matcher": "startup|resume",
    "hooks": [{
      "type": "command",
      "command": "/Users/<you>/.claude/scripts/vault-session-start.sh",
      "timeout": 10
    }]
  }],
  "Stop": [{
    "hooks": [{
      "type": "command",
      "command": "/Users/<you>/.claude/scripts/vault-session-end.sh",
      "async": true,
      "timeout": 30
    }]
  }],
  "PostCompact": [{
    "hooks": [{
      "type": "command",
      "command": "/Users/<you>/.claude/scripts/vault-session-start.sh",
      "timeout": 10
    }]
  }]
}

Globalny ~/.claude/CLAUDE.md

Auto-loaded w każdej sesji Claude Code (każdy projekt, każdy katalog). Powiedz LLM o vault sekcją "Obsidian Vault":

  • Lokalizacja vault (cross-projekt - jeden niezależnie od cwd)
  • Co zapisywać auto: decisions, people, projects, mail-out, learnings - immediate sync rule
  • Wikilinks zawsze, frontmatter YAML zawsze
  • NIGDY nie wpisuj secretów do vault
  • Reference do <vault>/CLAUDE.md dla szczegółów

Setup krok po kroku (12 kroków)

  1. Install Obsidian + utwórz vault Pobierz z obsidian.md (free, macOS/Windows/Linux). Utwórz vault np. ~/Documents/Obsidian/<nazwa>/. Zero konta wymagane (konto tylko dla Obsidian Sync $4/mies).
  2. Subskrypcja Claude Code Pro lub Max ($20/$200/mies) z claude.com. NIE używaj third-party narzędzi typu OpenCode/Cline z Twoim OAuth - automatyczny ban.
  3. Git init vault + private repo na GitHub 
    cd ~/Documents/Obsidian
git init -b main
gh repo create <username>/Obsidian --private --source=. --push
    .gitignore z exclusion: .DS_Store, .obsidian/workspace*.json, .obsidian/cache/, .obsidian/plugins/, *.key, *.env, secrets/.
  4. Install AgriciDaniel claude-obsidian do vault 
    git clone https://github.com/AgriciDaniel/claude-obsidian /tmp/co
cp -R /tmp/co/skills <vault>/.claude/
cp -R /tmp/co/commands <vault>/.claude/
    Daje 11 skills (zawiera Kepano obsidian-skills) + 4 slash commands (/wiki, /save, /canvas, /autoresearch).
  5. Install community plugins w Obsidian Settings → Community plugins → Browse:
    • Templater - install - dynamic templates
    • Dataview - install - SQL-like queries
    • Tasks - install - TODO management
    • Obsidian Git - install - auto-commit/push
    Linki obsidian:// działają jeśli masz Obsidian zainstalowany - klik = direct install.
  6. Skonfiguruj Obsidian Git Settings → Obsidian Git: backup interval 5 lub 10 min, Auto push ON, Pull on startup OFF (jeśli 1 urządzenie).
  7. Utwórz schema document vault/CLAUDE.md Twój kontrakt z LLM jak utrzymywać wiki. Wzór wyżej w sekcji "Sample vault/CLAUDE.md". Definiuje folder structure, konwencje, kiedy zapisywać auto.
  8. Utwórz vault/log.md i vault/hot.md log.md = append-only chronological z parseable prefixes ## [YYYY-MM-DD HH:MM] type | topic.
    hot.md = recent context cache (max 500 słów, overwritten każdą sesją).
  9. Utwórz skrypty bash hooków Pliki ~/.claude/scripts/vault-session-start.sh i vault-session-end.sh (kod wyżej w sekcji "Raw kod skryptów hooków"). chmod +x obie.
  10. Setup hooki w ~/.claude/settings.json Dodaj hooks block (kod wyżej). Zachowaj istniejące hooki - merge, NIE nadpisuj. Zwalidauj JSON: jq '.' ~/.claude/settings.json.
  11. Update ~/.claude/CLAUDE.md z sekcją "Obsidian Vault" Globalny CLAUDE.md auto-loaded w każdej sesji - mówi LLM gdzie jest vault i co tam zapisywać. Bez tego LLM w innym projekcie nie wie że vault istnieje.
  12. Test + restart Claude Code Hooki aktywne dopiero po restart sesji (settings watcher). Otwierasz nową sesję - SessionStart hook czyta hot.md, masz cross-projekt context od pierwszej wiadomości.

Daily flow (po setupie)

  1. Otwierasz Claude Code w dowolnym projekcie
  2. SessionStart hook wstrzykuje hot cache → LLM zna kontekst
  3. Pracujesz - LLM zapisuje ważne decyzje/maile/learnings do vault automatycznie
  4. /exit → Stop hook commituje + pushuje do GitHub
  5. Vault rośnie sam w tle, czytasz gdy szukasz

Co dalej po setupie - pierwsze use cases

  1. Otwórz Graph view w Obsidian (Cmd+G macOS, Ctrl+G Windows/Linux) - zobacz visualnie knowledge graph swoich notatek (people ↔ projects ↔ decisions cross-linked)
  2. Spróbuj /wiki w Claude Code - AgriciDaniel command sprawdza setup vault, scaffolduje brakujące struktury, zaproponuje pierwsze use case
  3. Pierwsza decyzja w sesji - powiedz Claude'owi explicit "to jest ważna decyzja, zapisz do vault" - obserwuj jak decisions/YYYY-MM-DD <topic>.md się tworzy auto + entry w log.md
  4. Pierwsze daily note - rano "Claude, wygeneruj mi dzisiejszą daily note z pending tasks z wczoraj" - tworzy się daily/YYYY-MM-DD.md
  5. Po tygodniu: /lint - health check vault per Karpathy: orphans, dead links, contradictions, missing cross-references
  6. Customize templates - vault/templates/ ma startery (mail-draft, project-setup, decision-log) - dostosuj pod swój workflow
  7. Po miesiącu - zerknij na log.md i graph view. Zobaczysz gdzie kompletujesz wiedzę, gdzie luki. Vault staje się Twoim "second brain" o którym mówił Karpathy.

Caveats / Limitations

Co ten setup NIE robi / czego nie obsługuje:
  • macOS-centric - skrypty bash i ścieżki działają na macOS. Linux wymaga drobnej adaptacji (stat ma inny syntax). Windows wymaga WSL lub PowerShell port.
  • Wymaga Claude Code subscription ($20/$200/mies) - bez tego brak hooków + brak slash commands.
  • Wymaga technical comfort z terminal, git, edycją JSON. Nie jest "click-and-go" jak Notion.
  • Mobile = osobne - Obsidian mobile apka istnieje (iOS/Android) ale wymaga Obsidian Sync $4/mies dla seamless sync. Bez tego: tylko desktop.
  • Hooki nie aktywują się od razu w sesji w której zmieniłeś settings.json - settings watcher widzi hooks od następnej sesji. Restart Claude Code po setupie.
  • Vault content jest curated, nie auto-everything. Hot.md wymaga okresowego review (Karpathy ostrzeżenie - "don't write-only wiki").
  • Privacy - vault commitowany do GitHub. Nawet private repo = data leaves your machine. Zero secrets w vault (App Passwords w Keychain, tokens w ~/mcp-keys/).
  • Brak undo dla auto-commits - Stop hook commituje + pushuje. Jeśli sesja zrobiła coś niechcianego do vault, trzeba ręcznie git revert.

FAQ

Czy działa na Windows?

Częściowo. Obsidian, Claude Code, AgriciDaniel skille, slash commands, Obsidian Git - wszystko działa cross-platform. Bash hooki wymagają WSL (Windows Subsystem for Linux) albo trzeba przepisać na PowerShell. Path syntax też się różni (C:\Users\ vs ~/).

Mogę użyć innego LLM (Gemini, GPT, Grok)?

Częściowo. Vault jako pliki .md z Karpathy patternem działa z każdym LLM. Ale konkretne narzędzia (AgriciDaniel skills, /wiki commands, hooki Claude Code) są Claude Code-specific. Inne CLI (np. Gemini CLI, Codex) mają własne hook systems.

AgriciDaniel deklaruje multi-model support (Gemini, Codex, Cursor, Windsurf) ale Claude integration jest najgłębsza.

Co z multi-device sync?

Trzy opcje:

  • Git push/pull (free) - manual git pull na każdym urządzeniu, działa ale wolniejsze
  • iCloud Drive (free dla Apple users) - vault folder w iCloud, automatyczny sync, mobile via Obsidian iOS apka też respektuje
  • Obsidian Sync ($4-8/mies) - end-to-end encrypted, fast, conflict resolution UI, multi-device. Polecane dla power users.
Ile to faktycznie kosztuje?

Minimum: $0/mies jeśli już masz Claude Code Pro. Obsidian free + GitHub free private repo + wszystkie skille open source.

Z Claude Code Pro: $20/mies (większość tematów osobno).

Z Claude Code Max + Obsidian Sync mobile: $204/mies (przy Max 5x) lub $204/mies + $4 Sync.

Wszystkie narzędzia w setupie (Obsidian, Templater, Dataview, Tasks, Obsidian Git, Kepano skills, AgriciDaniel) są open source / free.

Czy jest ryzyko bana Anthropic?

Ten setup = LOW risk. Hooki bash są deterministyczne, NIE wywołują Claude API (więc nie generują token volume który wygląda jak harness pattern). Skille i slash commands są oficjalnymi mechanizmami Claude Code.

Czego NIGDY nie robić: third-party harness (OpenCode, Cline, Cursor) z Twoim Claude OAuth (instant ban), recursive hook który wywołuje claude CLI w pętli, autonomous cron który puszcza claude -p "..." 24/7 bez user input, account sharing.

Anthropic zbanował 1.45 mln kont w H2 2025, apele tylko 3.3% sukcesu. Trzymaj się oficjalnych features = bezpieczny.

Co jeśli vault urośnie do 1000+ notatek?

Obsidian skaluje OK do ~10k notatek lokalnie. Search może zwolnić, graph view robi się chaotic. Wtedy:

  • Włącz Smart Connections plugin + smart-connections-mcp - lokalne embeddings (bge-micro-v2, free), semantic search dla Claude
  • Lint regularly - /lint per Karpathy - usuwa orphans, deadl inks, stale claims
  • Archive folder - przenoszenie starych notatek do archive/ żeby graph view był czystszy
Mogę to zrobić z private repo?

Vault repo - TAK, polecane (private). Ale jeśli chcesz publikować vault jako stronę web (Obsidian Publish $10/mies), wymagana publiczna ekspozycja zawartości.

Github Pages dla rendered vault - publishing tutoriali / public knowledge - wymaga PUBLIC repo (free tier) lub PUBLIC repo + GitHub Pro ($4/mies dla private Pages).

Backup strategy?

Trzy warstwy:

  • Git remote = automatic backup po każdym commit (Obsidian Git plugin co 5-10 min + Stop hook po sesji)
  • Time Machine (macOS) - lokalny backup całego dysku, codzienny snapshot
  • Drugi clone na innym Macu (jeśli masz) - git clone z GitHub origin, periodic git pull

Vault to pliki .md - nawet jeśli wszystko padnie, otwierasz w dowolnym text editor i czytasz.

Efekt

  • Cross-projekt memory - sesja w projekcie A widzi wszystko co działo się w projekcie B
